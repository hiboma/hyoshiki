# ADR-0003: データスキーマとふりがなの分離

## ステータス

採用

## コンテキスト

子供向けアプリのため、標識名に漢字が含まれる場合はふりがなを表示する必要があります。
また、読み上げ用テキストは表示用テキストと異なる区切りが必要です。

## 決定

`signs.json` のスキーマに `reading` と `speech` フィールドを設けて、
表示・読み上げ・データの各用途を分離します。

### スキーマ

```json
{
  "id": "305",
  "name": "二輪の自動車・原動機付自転車通行止め",
  "category": "規制標識",
  "image": "data/signs/305.png",
  "reading": "にりんのじどうしゃ・げんどうきつきじてんしゃつうこうどめ",
  "speech": "にりんのじどうしゃ、げんどうきつきじてんしゃ、つうこうどめ"
}
```

### 各フィールドの用途

| フィールド | 用途 | 必須 |
|---|---|---|
| `name` | カード表示の漢字テキスト | 必須 |
| `reading` | ルビ表示 (ひらがな) | 必須 |
| `speech` | Web Speech API への入力 (句読点で区切り) | 任意 |

### 読み上げの優先順位

```
speech > reading > name
```

`speech` が未定義の場合は `reading` を使います。
短い名前 (目安: 10 文字以下) には `speech` を設定しません。

### `speech` の変換ルール

1. `reading` の「・」を読点「、」に変換します
2. 長い複合語は意味の区切りに読点を追加します
3. 不自然にならない範囲に留めます

## 代替案

### A. `reading` に句読点を含めて兼用する

ルビ表示に句読点が現れるため、不採用としました。

### B. コード側で自動変換する

「・」→「、」は自動化できますが、長い複合語の区切りは文脈依存であり、
自動化が困難なため不採用としました。
